🧭 DIRECT: When and Where Should You
Allocate Test-Time Compute in Embodied Planners?

Anonymous Authors
Hardware Demos Abstract

DIRECT routes each task to the cheapest capable planner — matching a stronger model's success at up to 65% lower latency on a physical Franka arm.

Abstract

VLMs are increasingly used as high-level planners for embodied agents, and a common strategy is to scale test-time compute for better capability. But that compute buys uneven, often diminishing gains while adding latency, tokens, and FLOPs. We argue that choosing when and where to spend it is what brings frontier performance to the real world. DIRECT is a routing framework that uses multimodal scene context to allocate compute per task, improving the success–cost frontier over any fixed model. Across three axes — chain-of-thought depth, model size, and memory history — we show test-time compute is not a uniform lever: each yields a distinct capability gain. On VLABench, RoboMME, and a physical Franka arm, DIRECT matches or beats a stronger model's success at up to 65% lower latency.

Three Axes, Three Distinct Gains

  Chain-of-Thought

On 44% of VLABench tasks, a non-thinking model matches or beats Thinking at <2% of the latency. Reasoning pays off only sometimes.

  Model Size

Scaling 2B→235B traces no clean curve. Size mainly governs the breadth of skills a planner can command — useful only when a task needs them.

  Memory

Memory helps history-dependent tasks but hurts elsewhere. No single architecture dominates across difficulty.

How DIRECT Works

From a scene image and instruction, a lightweight router picks the best planner from a fixed pool. It encodes the image (frozen SigLIP) and text (frozen BGE-M3) and predicts the quality–cost tradeoff per planner — adding just 20–50 ms, negligible against any VLM (>1 s). The router runs once per task (or per step for multi-stage tasks), and deploys zero-shot on hardware.

Over 270k simulated routing decisions and 245 hardware trajectories.

Chain-of-Thought

Escalates to thinking only where it helps — near-oracle, beating OOD and random routing.

Model Size

Routing over 2B–235B adds +5.1 pts while cutting 32.4 s of latency at the 32B tier.

Memory

Near-oracle success while processing far fewer frames — paying recall cost only when a task needs it.
